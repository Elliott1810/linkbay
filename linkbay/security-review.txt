SECURITY REVIEW — Linkbay
=========================

1. HARDCODED SECRETS
--------------------
WARN: Hardcoded fallback session secret "linkbay-dev-secret-change-in-prod" in server/index.ts line 67.
  The code does guard against missing SESSION_SECRET in production (process.exit(1)), so this fallback
  only activates in development. Still risky if NODE_ENV is accidentally unset in a staging deploy.
  Recommendation: remove the fallback string entirely and always require the env var.

OK: No API keys, tokens, or real credentials found hardcoded in .ts/.tsx source files.

2. SQL INJECTION
----------------
OK: All SQL in storage.ts uses parameterised prepared statements (sqlite.prepare(...).run/get/all with ?
  placeholders). Drizzle ORM is used for all other queries. No raw string interpolation found in SQL.

3. MISSING AUTH CHECKS
----------------------
OK: All sensitive owner routes (pages CRUD, links CRUD, leads, analytics, dashboard, account) are
  gated with requireAuth middleware and assertOwnsPage ownership checks.

WARN: GET /api/pages/polls/:pollId/votes (line 492) has no auth requirement. Poll vote counts are
  public, which appears intentional, but there is no rate limiting on this endpoint. A client could
  poll it at high frequency to scrape vote activity.

4. SENSITIVE DATA IN API RESPONSES
-----------------------------------
BLOCK: GET /api/pages/public/:username returns the full Page object including the ownerEmail field
  (server/routes.ts line 137: res.json({ page, links })). The Page type (shared/schema.ts line 49)
  contains owner_email. Any anonymous visitor to a public profile page receives the page owner's
  registered email address in the API response. Strip ownerEmail (and ownerName if not needed) from
  the public-facing response object before launch.

WARN: POST /api/builder/create (line 483) leaks internal error message details in production:
  res.status(500).json({ error: "Server error", details: (err as any).message })
  This is the only 500 handler that exposes err.message. Internal DB errors, file paths, or stack
  fragments could be exposed. Remove the `details` field from the production 500 response.

OK: Auth endpoints (/api/auth/signup, /api/auth/login, /api/auth/me) never return passwordHash.
  User objects in responses are explicitly projected to { id, email, name }.

5. CORS MISCONFIGURATION
------------------------
OK: No cors() middleware is imported or applied in server/index.ts or server/routes.ts. The `cors`
  package appears in build.ts allowlist (a legacy artifact) but is not used. The app relies on
  same-origin delivery (Vite/Express serve both frontend and API on the same port), so no CORS
  headers are emitted. This is correct for a same-origin SaaS app.

6. SESSION COOKIE CONFIGURATION
---------------------------------
OK: In production, cookie name is "__Host-sid" which enforces Secure + path=/ at the browser level.
OK: httpOnly: true — cookie is not accessible to JavaScript.
OK: secure: true in production.
OK: sameSite: "none" in production (required for cross-origin reverse proxy); paired with secure:true
  this is valid. sameSite: "lax" in development.
OK: SESSION_SECRET missing in production causes immediate process.exit(1) — fail-fast is correct.
WARN: Session maxAge is 7 days with no sliding expiry (resave: false, saveUninitialized: false).
  Sessions are never explicitly rotated after privilege changes. Acceptable for most SaaS but worth
  noting; consider regenerating the session ID on login (req.session.regenerate).

7. eval() AND dangerouslySetInnerHTML
---------------------------------------
OK: No use of eval() found in any .ts/.tsx source file.

WARN: client/src/components/ui/chart.tsx line 81 uses dangerouslySetInnerHTML to inject CSS custom
  properties into a <style> tag. The injected content is built from the ChartConfig prop (color
  strings and CSS variable values). This config comes from internal dashboard code, not from
  user-submitted text, so there is no direct XSS path. However, if chart config is ever populated
  from user data (e.g. page analytics labels), CSS injection would be possible. Add a CSS hex-color
  regex guard on the `color` value before interpolation as a defence-in-depth measure.

8. ADDITIONAL FINDINGS
-----------------------
WARN: Login endpoint returns different error messages for "no account with that email" vs "incorrect
  password" (routes.ts lines 59, 61). This allows user enumeration: an attacker can determine
  whether an email is registered by observing which error is returned. Unify to a single message
  such as "Invalid email or password."

WARN: PATCH /api/leads/:id/status (lines 354–363) accepts any string value for `status` with no
  validation against the allowed enum ("new" | "contacted" | "closed"). An authenticated owner
  could set an arbitrary status string. Add a z.enum(["new", "contacted", "closed"]) check.

WARN: Content-Security-Policy is explicitly disabled (server/index.ts line 20:
  contentSecurityPolicy: false). This is noted as being for Vite HMR flexibility in development,
  but it should be re-enabled with an appropriate policy for the production build, which does not
  use Vite HMR. Without CSP, XSS payloads (if any were injected) have no browser-level mitigation.

OK: URL scheme validation for links (http/https/mailto/tel) is enforced both in the Zod schema
  (shared/schema.ts) and in the PATCH /api/links/:id handler, blocking javascript: XSS via link URLs.

OK: Rate limiting (authLimiter: 20 req / 15 min) is applied to /api/auth/signup, /api/auth/login,
  and /api/builder/create. Enforced only in production (NODE_ENV=production).

OK: Username format is validated with /^[a-z0-9-]{3,40}$/ regex on both create paths, preventing
  path traversal or injection via the URL slug.

OK: Page update (PATCH /api/pages/:id) uses an explicit field allowlist via Zod, preventing
  mass-assignment of ownerEmail, viewCount, published, etc.

SUMMARY
-------
Blockers before public launch: 1
  - ownerEmail exposed in public page API response

Warnings to address: 8
  - Hardcoded dev secret fallback
  - Internal error details leaking in /api/builder/create 500 response
  - Poll vote endpoint lacks rate limiting
  - No session ID regeneration on login
  - User enumeration via distinct login error messages
  - Lead status update accepts arbitrary string (no enum validation)
  - CSP disabled globally including in production
  - dangerouslySetInnerHTML in chart.tsx on chart config color values (low risk, defence-in-depth)
